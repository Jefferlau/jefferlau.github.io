JHipster 学习笔记(四) - Common 应用程序属性文件

来自官网

JHipster 生成一个 Spring Boot 应用程序，可以使用标准的 Spring Boot 属性机制进行配置。

这些属性由 JHipster 在生成时配置，并且在开发和生产模式中通常具有不同的值：在 Profiles 文档 中了解更多信息。

在 JHipster 应用程序中，有三种属性：

  1. Spring Boot 标准应用程序属性
  2. JHipster 应用程序属性
  3. 应用程序特定的属性

Spring Boot 标准应用程序属性

像所有 Spring Boot 应用程序一样，JHipster 允许您配置任何标准的 Spring Boot 应用程序属性。

JHipster 应用程序属性

JHipster 提供了特定的应用程序属性，它们来自 JHipster服务器端库。这些属性是所有 JHipster 项目的标准配置，但其中一些属性仅在您构建应用程序时选择的方式工作：例如，jhipster.cache.hazelcast 键只有在选择 Hazelcast 作为 Hibernate 二级缓存时才起作用。

这些属性使用 io.github.jhipster.config.JHipsterProperties 类进行配置。

应用程序特定的属性

强烈建议您生成的应用程序也拥有自己的 Spring Boot 属性。 因为这允许应用程序的类型安全配置，以及 IDE 中的自动完成和文档功能。

JHipster 已经在配置包中生成了一个 ApplicationProperties 类，它已经被预先配置了，它已经在底层的 application.yml， application-dev.yml 和 application-prod.yml 文件中被记录下来了。 所有你需要做的就是编写你自己的具体属性。

JHipster 学习笔记(三) - 管理 Profiles

发表于 2017-06-12 | 分类于 JHipster | | 阅读次数

来自官网

Profiles

JHipster 提供两种 Spring profiles :

  • dev for development: 它侧重于易于开发和生产力
  • prod for production: 它侧重于性能和可扩展性

这些配置文件有两种不同的配置：

  • Maven / Gradle 配置文件在构建时使用。 例如./mvnw -Pprod package或./gradlew bootRepackage -Pprod将打包生产环境应用程序。
  • Spring 配置文件在运行时工作。一些 Spring beans 依赖不同配置文件其行为会有所不同。

Spring profiles 由 Maven / Gradle 设置，所以我们在两种方法之间保持一致：您将在 Maven / Gradle 和 Spring 同时拥有一个 prod profile。

注意：Spring profiles 文件用于配置 JHipster 应用程序属性，因此您应该有兴趣阅读 应用程序属性文件。

JHipster 默认使用 dev profile

如果您在没有 Maven / Gradle 的情况下运行应用程序，请启动 “Application” 类（您可以通过右键单击从IDE轻松运行）。

如果使用 Maven 运行应用程序，请运行./mvnw以使用 Maven 包装，或者使用mvn来使用自己安装的 Maven。

如果您使用 Gradle 运行应用程序，请运行./gradlew以使用我们的 Gradle 包装，或者使用gradle来使用自己安装的Gradle。

当使用 Angular 2+ 时，如果需要执行干净的运行，使用 webpack 编译启用 dev profile 配置文件，您可以传递 webpack 参数如下

./mvnw -Pdev,webpack 或 ./gradlew -Pdev -Pwebpack

在生产环境 JHipster 运行 prod profile

在生产环境可以直接使用 Maven 或 Gradle 运行 JHipster：

  • Maven, 运行 ./mvnw -Pprod (或 mvn -Pprod)
  • Gradle, 运行 ./gradlew -Pprod (或 gradle -Pprod)

如果想将应用打包成可执行 WAR 文件，只需要给 Maven 或 Gradle 一个 profile，例如：

  • Maven, run ./mvnw -Pprod package (或 mvn -Pprod package)
  • Gradle, run ./gradlew -Pprod bootRepackage (或 gradle -Pprod bootRepackage)

当使用 WAR 文件运行生产环境应用时，默认使用和打包时相同的 profile(s)。如果想覆盖这个配置，可以在 VM 参数中显式提供一种替代方法：

Spring profiles 开关

JHipster 带有三个额外的配置文件用作开关：

  • swagger 开启 swagger
  • no-liquibase 关闭 liquibase

这些可以与 dev 和 prod 配置文件一起使用。 请注意在默认情况下，在 application.yml 中设置 spring.profiles.include 属性，使swagger在 prod 中禁用并且在 dev 中启用。

swagger 和 no-liquibase 仅用在运行时:

  • 在 IDE 中，使用 spring.profiles.active=dev,no-liquibase运行主应用程序类（请注意，需要显式地包含 dev 或 prod 配置文件）
  • 使用打包应用程序：./java -jar jhipster-0.0.1-SNAPSHOT.war --spring.profiles.active=prod,no-liquibase

使用 Maven，也可以直接使用这些配置文件：

  • ./mvnw -Pprod,swagger,no-liquibase
  • ./mvnw -Pdev,no-liquibase

使用 Gradle, 也可以直接使用这些配置文件：

  • ./gradlew -Pprod -Pswagger -Pno-liquibase
  • ./gradlew -Pno-liquibase

运行 Yarn

项目直接依赖关系被配置到package.json中，但是传递依赖关系被定义在yarn.lock文件，它在运行yarn install时生成。

建议将yarn.lock放到源代码版本控制中，以便项目组的所有团队成员的所有依赖关系都具有相同的版本。
再次运行yarn install将重新生成具有最新版本的传递依赖关系的yarn.lock。

数据库更新

如果增加或修改了 JPA entity，将需要更新数据库。

JHipster使用Liquibase来管理数据库更新，并将其配置存储在/src/main/resources/config/liquibase/目录中。 使用 Liquibase 有3种方法：使用实体子生成器(entity sub-generator)，使用 liquibase：diff Maven目标，或手动更新配置文件。

使用 Maven liquibase:diff 更新数据库

如果选择在开发中使用 MySQL，MariaDB 或 PostgreSQL，则可以使用 ./mvnw liquibase:diff目标自动生成 changelog。

如果您使用基于磁盘的持久性运行 H2，这个尚不能完美工作，但您可以开始尝试使用它（并向官方发送反馈！）。

Liquibase Hibernate 是一个在 Maven pom.xml 中配置的插件，与 Spring application.yml 文件无关，因此如果更改了默认设置（例如，更改了数据库密码），则需要修改这两个文件。

这是开发工作流程：

  • 修改 JPA entity (增加字段 [field], 关系 [relationship], 等。)
  • 编译应用 (编译 Java 代码)
  • 运行 ./mvnw liquibase:diff (或 ./mvnw compile liquibase:diff)
  • 新的 “change log” 被创建在 src/main/resources/config/liquibase/changelog 文件夹
  • 查看此 “change log” 并将其添加到您的 src/main/resources/config/liquibase/master.xml 文件, 在下次运行应用程序时应用它

如果您使用 Gradle 而不是 Maven，则可以通过运行 ./gradlew liquibaseDiffChangelog来使用相同的工作流程，如果需要，可以更改liquibase.gradle中的数据库配置。

手工修改 change log 更新数据库

如果您喜欢（或需要）手动进行数据库更新，则以下是开发工作流程：

  • 修改 JPA entity (增加字段 [field], 关系 [relationship], 等。)
  • 创建新的 “change log” 在src/main/resources/config/liquibase/changelog 文件夹。该目录中的文件以其创建日期（以 yyyyMMddHHmmss 格式）为前缀，然后具有描述其作用的标题。 例如，20141006152300_added_price_to_product.xml是一个好名字。
  • 添加 “change log” 文件到 src/main/resources/config/liquibase/master.xml 文件, 在下次运行应用程序时应用它

如果想了解更多的 Liquibase 使用信息，请移步http://www.liquibase.org。

JHipster 简介

JHipster 是一个用于创建、开发和部署 Spring Boot + Angular Web 应用和 Spring 微服务的开发平台。

JHipster 致力于创建完整且现代的统一 Web 应用或微服务架构。

  • 高性能、健壮的使用 Spring Boot 的服务器端 Java 栈
  • 使用 Angular 和 Bootstrap 的时尚现代地前端
  • 使用 JHipster Registry, Netflix OSS, ELK 栈和 Docker 的健壮的微服务架构
  • 使用 Yeoman，Webpack / Gulp和Maven / Gradle 构建应用程序的强大工作流程

📘 The JHipster Mini-Book v2.0 by Matt Raible

单页面应用:

  • AngularJS v1.x or Angular v2+
  • 响应式页面设计 Twitter Bootstrap
  • HTML5 Boilerplate
  • 兼容 IE11 和其他现代浏览器
  • 完整的国际化支持
  • 可选 Sass 支持 CSS 设计

  • Spring Boot 简化应用配置
  • Maven 或 Gradle 构建，测试和运行应用
  • “development” 和 “production” profiles (支持 Maven 和 Gradle)
  • Spring Security
  • Spring MVC REST + Jackson
  • 可选 Spring Websocket 支持 WebSocket
  • Spring Data JPA + Bean 验证
  • 使用 Liquibase 实现数据库更新
  • Elasticsearch 支持对数据库的搜索
  • 支持像 MongoDB 这样的 document-oriented NoSQL 数据库如果不愿意使用 JPA
  • 支持像 Cassandra 这样的 column-oriented NoSQL 数据库如果不愿意使用 JPA

  • 可选使用 hazelcast 实现 HTTP session 集群
  • 优化的静态资源 (gzip filter, HTTP cache headers)
  • 日志管理使用 Logback, 可在运行时配置
  • 使用 HikariCP 连接池以获得最佳性能
  • 可以构建成一个 WAR 文件或一个可执行的 JAR 文件
  • 支持所有重要的云服务供应商: AWS, Cloud Foundry, Heroku, Kubernetes, Docker 等等

Spring Boot 外化配置及启动方式不同引发的问题

发表于 2017-04-23 | 分类于 Java ， Spring Boot | | 阅读次数

近日有网友问我关于 Spring Boot 外化配置相关问题，但是对方始终跟我说配置未生效，并将配置文件发来，检查后也未发现问题，直到对方说明他使用 java -jar 的方式启动且执行时并不在 jar 所在目录，我才明白他的配置为什么不生效，下面对 Spring Boot 做一些浅显的解读，并说明他的配置不生效的原因。

先简单说一下 Spring Boot 的外化配置。

我们先看一下 Spring Boot 配置生效的顺序

  1. Home 目录下的 Devtools 全局设置属性（~/.spring-boot-devtools.properties，当 devtools 激活时）。
  2. 测试用例上的 @TestPropertySource 注解。
  3. 测试用例上的 @SpringBootTest#properties 注解。
  4. 命令行参数。
  5. 来自 SPRING_APPLICATION_JSON 的属性（环境变量或系统属性中内嵌的内联 JSON ）。
  6. ServletConfig 初始化参数。
  7. ServletContext 初始化参数。
  8. 来自于 java:comp/env 的 JNDI 属性。
  9. Java 系统属性（System.getProperties()）。
  10. 操作系统环境变量。
  11. RandomValuePropertySource，仅包含 random.* 中的属性。
  12. 没有打进 jar 包的 Profile-specific 应用属性（application-{profile}.properties 和 YAML 变量）。
  13. 打进 jar 包中的 Profile-specific 应用属性（application-{profile}.properties 和 YAML 变量）。
  14. 没有打进 jar 包的应用配置（application.properties 和 YAML 变量）。
  15. 打进 jar 包中的应用配置（application.properties 和 YAML 变量）。
  16. @Configuration 类上的 @PropertySource 注解。
  17. 默认属性（使用 SpringApplication.setDefaultProperties 指定）。

这个生效顺序至少要在脑海中有个印象，才能在应用开发的生命周期中少走弯路。

下面以应用名称为 spring-boot 来作为示例，预设此应用打包后得到名称为 spring-boot.jar的可执行 jar。示例中WEB 服务的端口默认为8080，外化配置要完成的功能是修改应用启动时提供 WEB 服务的端口为8082。
外化的 application.yml 内容如下：
Configuration server: port: 8082
那么回到我们一开始遇到的问题，如果我们将可执行 jar 放在 /opt/target目录，那么这个 application.yml 文件要放在哪里，启动时配置才会生效？

可执行 jar 有多种启动方式，比如安装成 service、 直接执行、java -jar xx.jar 等等，这里不一一列举。我们这里只说直接执行和java -jar这两种方式，这两种方式放到 Docker 执行都比较方便。

我们先将 application.yml 文件放在 spring-boot.jar 同一个目录，即/opt/target/application.yml。

我们登陆服务器后进入/opt目录。

  • 先执行target/spring-boot.jar（这里忽略执行权限），发现 WEB 端口已经按照期望修改成了8082，ctrl+c 关闭；

  • 再执行java -jar target/spring-boot.jar，这时发现 WEB 端口依然是8080，ctrl+c 关闭；

  • 进入到 target 目录再执行java -jar spring-boot.jar，这时发现 WEB 端口已经变成是8082，ctrl+c 关闭；

我们来看看为什么使用 java -jar 启动时，读取的 application.yml 文件为啥不是读 jar 文件所在目录的文件，打开 Spring Boot 读取配置文件的 ConfigFileApplicationListener.java源代码，我们看两个位置，一是类的文档第一段，二是110行DEFAULT_SEARCH_LOCATIONS变量的定义。

那么为什么直接执行 jar 或者 service 的方式启动时，都能加载到和 jar 同目录的 application.yml 文件呢？想知道这个答案，我们来看看 Spring Boot 的launch.script，很容易发现，这个脚本会到 jar 文件所在的目录，再执行 java -jar。

正确的放置位置是：

  • 当我们选用 java -jar 启动 Spring Boot 时，我们要将 application.yml 放在执行 java -jar 时，会话所在的目录。
  • 当我们选用直接执行时，application.yml 放在可执行 jar 同一个目录。

开发环境 MySQL 密码过期

发表于 2017-01-21 | 分类于 MySQL | | 阅读次数

开发环境 MySQL Server version: 5.7.10
今天开机连接本机 MySQL 突然报错，客户端无法连接了，于是使用命令行排查故障。

从命令行提示可以看到，需要重新设置密码，但是设置简单密码时又不符合密码安全策略，不想改密码而且本地开发环境，也不需要搞的这么复杂于是改下 MySQL 服务的配置，取消密码安全策略检查。
修改 /etc/my.cnf，在 [mysqld] 节点增加 validate-password=OFF，然后重启服务。再次设置密码即可。

/etc下没有 my.cnf 的话可以从 MySQL_HOME/support-files/ 里复制 my-default.cnf

validate-password 参数说明如下：

  • validate-password=ON/OFF/FORCE/FORCE_PLUS_PERMANENT:决定是否使用该插件(及强制/永久强制使用)。
  • validate_password_dictionary_file： 插件用于验证密码强度的字典文件路径。
  • validate_password_length： 密码最小长度。
  • validate_password_mixed_case_count： 密码至少要包含的小写字母个数和大写字母个数。
  • validate_password_number_count： 密码至少要包含的数字个数。
  • validate_password_policy： 密码强度检查等级，0/LOW、1/MEDIUM、2/STRONG。
  • validate_password_special_char_count： 密码至少要包含的特殊字符数。

其中关于 validate_password_policy 密码强度检查等级：
0/LOW：只检查长度。
1/MEDIUM：检查长度、数字、大小写、特殊字符。
2/STRONG：检查长度、数字、大小写、特殊字符字典文件。
