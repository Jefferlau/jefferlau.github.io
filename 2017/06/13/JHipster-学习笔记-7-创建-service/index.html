创建 service

介绍

与 “entity” sub-generator 相比，这是一个非常简单的 sub-generator。

它的目标是生成 Spring Service bean，这是应用程序的业务逻辑应该被编码的地方。

这将产生一个简单的 “BarService”：很少的代码，但他们通常会有很多问题。 我们尝试回答下面最常见的。

为什么不是 “entity” generator 生成的 Service 类？

我们在这里有两个主要的架构原理：

  • 我们不想创建无用服务：如果您需要的是数据库上的基本 CRUD，则不需要 Service Bean。 这就是为什么默认情况下，JHipster 不会生成它们。
  • 我们相信一个 Service bean 比 repository 更粗粒度。 Service bean 将使用多个 repository 来提供业务价值。 这就是为什么不能使用实体生成 Service bean。

Service Beans 应该使用接口吗?

简答：不使用。

如果想要很长的答案，在这里：

使用 Spring 的主要兴趣之一是 AOP。 这是允许 Spring 在您的 Bean 之上添加新行为的技术：例如，事务或安全性的工作。

为了添加这些行为，Spring 需要在你的类上创建一个代理，创建代理有两种方法：

  • 如果您的类使用接口，Spring 将使用 Java 提供的标准机制来创建动态代理。
  • 如果您的类不使用接口，Spring 将使用 CGLIB 即时生成一个新类：这不是一个标准的 Java 机制，但它与标准机制一样。

有些人也会认为接口编写测试更好，但是我们认为我们不应该修改测试的生产代码，而且所有新的 mocking 框架（如 EasyMock）都允许您在没有任何接口的情况下创建非常好的单元测试。

所以，最后我们发现 Service bean 的接口大都是没用的，这就是为什么我们不推荐它们（但我们让你选择生成它们）。

为什么要使用事务来获取 JPA 延迟关系?

默认情况下，JPA 使用一对多和多对多实体关系的延迟初始化。 如果您使用此默认配置，您可能会看到可怕的 LazyInitializationException：这意味着您尝试在事务之外使用未初始化的关系。

由于生成的 Service 类默认使用 @Transactional 注解，所有它的方法都是事务性的。 这意味着您可以在这些业务方法中获取所有必需的懒惰关系，而不需要任何LazyInitializationException。

提示：如果您不修改任何数据，则在方法上使用@Transactional(readOnly = true)。 这是一个不错的性能优化（Hibernate 不需要刷新其第一级缓存，因为我们不修改任何东西），以及一些 JDBC 驱动程序的质量增强（Oracle不会允许您发送 INSERT/UPDATE/DELETE 语句）

可以添加 security 到 Service Beans 吗?

可以! 只需在您的类或方法上添加 Spring Security 的 @Secured 注解，并使用提供的 AuthoritiesConstants 类限制对特定用户权限的访问。

可以监控 Service Beans 吗?

可以! 只需在要监视的方法上添加 Metrics 的 @Timed 注解。
