负载均衡

Nginx 提供了以下三种负载均衡机制：

  1. round-robin — （默认）请求以轮询的方式分发到应用服务器，如果后端服务器 down 掉，能自动剔除。
    weight 指定轮询几率，weight 和访问比率成正比，用于后端服务器性能不均的情况。
  2. least-connected — 下一个请求被分配到拥有最少活动连接数的服务器
  3. ip-hash — 使用一个哈希函数，基于客户端 IP 地址判断请求应该被分发到哪个服务器,这样每个访客固定访问一个后端服务器，可以解决 session 的问题。

在之前配置 Java 反向代理的时候 nginx.conf 里 upstream 就是用来定义被 Nginx 代理的服务器组

默认情况下，使用轮询机制，如果设置了权重，将会按照将会按照权重来分配响应的服务。

配置了 least-connected 的负载均衡机制的情况下，Nginx
会把新的请求发送到比较不繁忙的服务器。

ip_hash 机制下客户端 IP 地址被用作 hash key 来判断客户端请求应该发送到哪个服务器，这种方法保证了来自相同客户端的请求总是发送到相同服务器（如果服务器可用的话）

如果有必要把客户端绑定至特定服务器，则可使用ip-hash负载均衡机制。

升级应用程序

当新版本的 JHipster 发布时，JHipster 升级子生成器(upgrade sub-generator) 有助于将现有应用程序升级到新版本，而不会擦除您的更改。

这有助于：

  • 在现有应用程序中拥有最新的 JHipster 功能
  • 当有重要的错误修复或安全更新时，获取更改
  • 保留您的代码库中的更改，并轻松地将其与新生成的代码合并

在进行升级之前，请仔细阅读本页，了解升级过程的工作原理。

要求

对于这个子生成器，您需要从 http://git-scm.com 安装 git。

运行升级子生成器

进入应用根目录：

要升级您的应用程序，请键入：

以下是您可以使用的选项：

  • –verbose - 详细记录升级过程的每一步
  • –target-version=4.2.0 - 升级到目标版本，而不是最新的版本，如果一个项目落后了几个版本，这是非常有用的，需要逐级升级
  • –force - 即使没有新的 JHipster 版本，强制运行升级子生成器

升级过程的图形视图

以下以图形方式展示升级过程如何工作（请阅读以下部分以获得文字说明）：

（这张图片来自 JSFiddle ）

请注意，jhipster_upgrade 分支将在您的项目中创建为孤儿，尽管在上图上没有正确显示。

升级过程的逐步说明

以下是 JHipster 升级子生成器处理的步骤：

  1. 检查是否有新版本的 JHipster 可用（如果您使用 --force 不适用）。
  2. 检查应用程序是否已被初始化为 git 存储库，否则 JHipster 将为您初始化一个，并将当前代码库提交给主分支。
  3. 检查以确保存储库中没有未提交的本地更改。 如果发现未提交的更改，则进程将退出。
  4. 检查是否存在 jhipster_upgrade 分支。 如果没有，将创建一个分支：在“第一次升级中执行的特定步骤”部分中提供有关此步骤的详细信息。
  5. 检出 jhipster_upgrade 分支。
  6. 将 JHipster 升级到全球最新版本。
  7. 清理当前项目目录。
  8. 使用 jhipster --force --with-entities 命令重新生成应用程序。
  9. 将生成的代码提交到 jhipster_upgrade 分支。
  10. 将 jhipster_upgrade 分支合并回启动 jhipster upgrade 升级命令的原始分支。
  11. 现在你只需要解决合并冲突，如果有的话。

恭喜，您的应用程序现在已升级为最新版本的 JHipster

首次升级时执行的具体步骤

在 JHipster 升级子生成器的第一次执行时，为了避免擦除所有更改，将执行一些其他步骤：

  1. jhipster_upgrade 分支被创建为孤儿（没有父项）
  2. 整个应用程序生成（使用您当前的 JHipster 版本）。
  3. 在主分支 master 上进行块合并提交：在主分支 master 上的代码库上不进行任何更改; 这只是在 Git 中记录 master 的 HEAD 与当前 JHipster 最新版本 。

忠告

不要在 jhipster_upgrade 分支上提交任何内容。 该分支专用于 JHipster 升级子生成器：每次运行子生成器时，将创建一个新的提交。

管理关系

当使用 JPA 时，实体子生成器可以创建实体之间的关系。

介绍

关系仅在使用 JPA 时有效。 如果您选择使用 Cassandra 或 MongoDB，它们将不可用。

关系在两个实体之间工作，JHipster 将生成以下代码：

  • 在生成的 JPA 实体中管理这种关系
  • 创建正确的 Liquibase 变更日志，以便在数据库中体现关系
  • 生成 AngularJS 前端，以便可以在用户界面中以图形方式管理此关系

JHipster UML 和 JDL Studio

本页介绍如何使用标准命令行界面与 JHipster 建立关系。 如果要创建多个实体和关系，则可能更喜欢使用图形工具。

在这种情况下，有两个选项可用：

  • JHipster UML，它允许您使用 UML 编辑器。
  • 在线工具 JDL Studio，使用 domain-specific 语言创建实体和关系。

您可以通过运行jhipster import-jdl your-jdl-file.jh，使用import-jdl子生成器从 JDL 文件生成具有关系的实体。

可用关系

当我们使用 JPA 时，可以使用通常的一对多，多对一，多对多和一对一的关系：

  1. 双向一对多关系
  2. 单向多对一关系
  3. 单向一对多关系
  4. 两个实体之间两个一对多关系
  5. 多对多关系
  6. 一对一关系
  7. 单向一对一关系

提示： User实体

请注意，由 JHipster 处理的User实体具体可以做：

  • 与该实体的多对一(many-to-one)关系（Car可以与User具有多对一的关系）。 这将在您的新实体存储库中生成特定查询，因此可以实现一个常见的需求，在当前的 security 用户上过滤您的实体。 在生成的 AngularJS 客户端界面上，您将在Car中下拉选择一个User。
  • 与User实体的多对多(many-to-many)和一对一(one-to-one)的关系，但另一个实体必须是关系的所有者（一个 Team 可以与 User 有多对多的关系，但只有团队可以添加/删除用户，用户无法添加/删除团队）。 在 AngularJS 客户端 UI 中，您还可以在多选框中选择User。

使用 DTO

警告 这是 BETA 质量的一个新功能。 使用它自己承担风险！ 非常欢迎反馈问题！

介绍

默认情况下，JHipster 直接在其 REST 端点中使用其领域对象（通常是 JPA 实体）。 这有很多好处，主要是它使代码易于使用，理解和扩展。

但是，对于复杂的用例，您可能希望使用将由 REST 端点公开的数据传输对象（DTO）。 这些对象在领域对象之上添加了一个额外的层，并且针对 REST 层进行了专门的调整：它们的主要优点是可以聚合多个领域对象。

DTO 如何在 JHipster 工作

生成 JHipster 实体时，您还可以选择为此实体生成 DTO。 如果选择该选项：

  • 将生成一个 DTO，并将其映射到基础实体上。
  • 将聚合多对一关系，仅用于使用 AngularJS 显示它的 ID 和字段。 例如，与 User 实体的多对一关系将向 DTO 添加一个 userId 字段和一个 userLogin 字段。
  • 将会忽略非所有者方面的一对多关系和多对多关系： 这与实体工作方式相匹配（它们在这些字段上具有 @JsonIgnore 注释）。
  • 对于所有者方面的多对多关系： 它将从另一实体以Set的形式来使用 DTO。 因此，如果其他实体也使用 DTO，只能这样工作。

使用 MapStruct 映射 DTO 和实体

由于 DTO 看起来很像实体，所以需要一个解决方案来自动映射它们是一个常见的需求。

JHipster 中选择的解决方案是使用 MapStruct。 它是一个注解处理器，插入到 Java 编译器中，这将自动生成所需的映射。

我们发现它非常干净和高效，并且喜欢它不使用反射（这在使用时像在 mapper 中一样重要）。

配置 IDE for MapStruct

MapStruct 是一个注解处理器，因此它也应该被设置为在 IDE 编译项目时自动运行。

如果您使用 Maven，则需要在 IDE 中激活 IDE maven 配置文件。 Gradle 用户不需要应用任何 IDE 特定的设置。

激活配置文件的说明包括在 配置 IDE 中。

MapStruct 高级用法

MapStruct 映射器被配置为 Spring Bean，并支持依赖注入。 有一个小窍门是可以将 Repository 注入映射器，因此可以使用其 ID 从映射器获取托管的 JPA 实体。

这将产生一个简单的 “BarService”：很少的代码，但他们通常会有很多问题。 我们尝试回答下面最常见的。

为什么不是 “entity” generator 生成的 Service 类？

我们在这里有两个主要的架构原理：

  • 我们不想创建无用服务：如果您需要的是数据库上的基本 CRUD，则不需要 Service Bean。 这就是为什么默认情况下，JHipster 不会生成它们。
  • 我们相信一个 Service bean 比 repository 更粗粒度。 Service bean 将使用多个 repository 来提供业务价值。 这就是为什么不能使用实体生成 Service bean。

Service Beans 应该使用接口吗?

简答：不使用。

如果想要很长的答案，在这里：

使用 Spring 的主要兴趣之一是 AOP。 这是允许 Spring 在您的 Bean 之上添加新行为的技术：例如，事务或安全性的工作。

为了添加这些行为，Spring 需要在你的类上创建一个代理，创建代理有两种方法：

  • 如果您的类使用接口，Spring 将使用 Java 提供的标准机制来创建动态代理。
  • 如果您的类不使用接口，Spring 将使用 CGLIB 即时生成一个新类：这不是一个标准的 Java 机制，但它与标准机制一样。

有些人也会认为接口编写测试更好，但是我们认为我们不应该修改测试的生产代码，而且所有新的 mocking 框架（如 EasyMock）都允许您在没有任何接口的情况下创建非常好的单元测试。

所以，最后我们发现 Service bean 的接口大都是没用的，这就是为什么我们不推荐它们（但我们让你选择生成它们）。

为什么要使用事务来获取 JPA 延迟关系?

默认情况下，JPA 使用一对多和多对多实体关系的延迟初始化。 如果您使用此默认配置，您可能会看到可怕的 LazyInitializationException：这意味着您尝试在事务之外使用未初始化的关系。

由于生成的 Service 类默认使用 @Transactional 注解，所有它的方法都是事务性的。 这意味着您可以在这些业务方法中获取所有必需的懒惰关系，而不需要任何LazyInitializationException。

提示：如果您不修改任何数据，则在方法上使用@Transactional(readOnly = true)。 这是一个不错的性能优化（Hibernate 不需要刷新其第一级缓存，因为我们不修改任何东西），以及一些 JDBC 驱动程序的质量增强（Oracle不会允许您发送 INSERT/UPDATE/DELETE 语句）

可以添加 security 到 Service Beans 吗?

可以! 只需在您的类或方法上添加 Spring Security 的 @Secured 注解，并使用提供的 AuthoritiesConstants 类限制对特定用户权限的访问。

可以监控 Service Beans 吗?

可以! 只需在要监视的方法上添加 Metrics 的 @Timed 注解。

创建 entity

推荐在开始前先看看官方视频 video tutorial!

重要 如果您想要“live reload”您的 JavaScript / TypeScript 代码，则需要运行 gulp （适用于 JavaScript/AngularJS 1）或 yarn start（用于 TypeScript/Angular 2+）。 您可以从 在开发中使用 JHipster 了解更多信息。

介绍

创建应用程序后，您将需要创建实体(entity)。 例如，您可能需要创建“Author”和“Book”实体。 对于每个实体，您将需要：

  • 数据库表
  • Liquibase 变更集
  • JPA Entity
  • Spring Data JPA Repository
  • Spring MVC REST Controller, 具有基本的 CRUD 操作
  • Angular 路由器，组件和服务
  • HTML 视图
  • 集成测试，以验证一切都正常工作
  • 性能测试，看看是否都流畅运行

如果您有几个实体，您可能希望在它们之间建立关系。 为此，您将需要：

  • 数据库外键
  • 用于管理此关系的特定 JavaScript 和 HTML 代码

“entity” sub-generator 将创建所有必需的文件，并为每个实体提供一个 CRUD 前端（参见项目结构）。 可以通过运行 jhipster entity <entityName> - [options]来调用子生成器。 可以通过键入 jhipster entity --help 找到这些选项的参考

以下是支持的选项。

  • --table-name <table_name> - 默认情况下，JHipster 将根据您的实体名称生成一个表名，如果您希望具有不同的表名，可以传递此选项。
  • --angular-suffix <suffix> - 如果您希望所有 Angular 路由都有自定义后缀，您可以使用此选项传递。
  • --regenerate - 这将重新生成一个现有的实体，而不会提出任何问题。
  • --skip-server - 这将跳过服务器端代码，只会生成客户端代码。
  • --skip-client - 这将跳过客户端 diam，只会生成服务端代码。

JHipster UML 和 JDL Studio

本页介绍如何使用标准命令行界面创建 JHipster 的实体。 如果要创建多个实体，则可能更喜欢使用图形工具。

在这种情况下，有两个选项可用：

  • JHipster UML，它允许您使用 UML 编辑器。
  • JDL Studio，我们的在线工具，使用我们的领域专用语言 JDL 创建实体和关系。

如果使用 JDL Studio:

  • 您可以使用 import-jdl 子生成器通过运行 jhipster import-jdl your-jdl-file.jh 从 JDL 文件生成实体。
  • 如果要使用 JHipster UML 而不是 import-jdl 子生成器，则需要通过运行 npm install -g jhipster-uml 来安装它，然后运行 jhipster-uml yourFileName.jh。

Entity 字段

对于每个实体，您可以根据需要添加任意数量的字段。 您将需要输入字段名称及其类型，JHipster 将为您生成所有必需的代码和配置，从 Angular HTML 视图到 Liquibase 变更日志。

这些字段在您使用的技术中不能包含保留关键字。 例如，如果你使用 MySQL：

  • 您不能使用 Java 保留关键字（因为您的代码将无法编译）
  • 您不能使用 MySQL 保留关键字（因为您的数据库更新将失败）

字段类型

JHipster 支持许多字段类型。 此支持取决于您的数据库后端，因此我们使用 Java 类型来描述它们：Java String 在 Oracle 或 Cassandra 中存储不同，它是 JHipster 为您生成正确的数据库访问代码的优势之一。

  • String: Java 字符串。 它的默认大小取决于底层后端（如果使用 JPA，默认值为 255），但是可以使用验证规则（例如设置max大小为 1024）进行更改。
  • Integer: Java 整数.
  • Long: Java 长整型.
  • Float: Java 单精度浮点数.
  • Double: Java 双精度浮点数.
  • BigDecimal: java.math.BigDecimal 对象，当您想要精确的数学计算（通常用于财务操作）时使用。
  • LocalDate: java.time.LocalDate 对象，用于在 Java 中正确管理日期。
  • Instant: java.time.Instant 对象，用于表示时间戳，即时间线上的瞬时点。
  • ZonedDateTime: java.time.ZonedDateTime 对象，用于表示给定时区中的本地日期时间（通常为日历约会）。请注意，时区不是 REST 支持的，也不是持久层支持的，所以您应该最有可能使用 Instant。
  • Boolean: Java 布尔值。
  • Enumeration: Java 枚举对象。 当选择此类型时，子生成器将询问您在枚举中需要什么值，并且它将创建一个特定的枚举类来存储它们。
  • Blob: Blob 对象，用于存储一些二进制数据。 当选择此类型时，子生成器将询问您是否要存储通用二进制数据，图像对象或 CLOB（长文本）。 图像将专门在角度处理，因此它们可以显示给最终用户。

验证

可以为每个字段设置验证。 根据字段类型，将提供不同的验证选项。

验证将自动生成：

  • HTML 视图，使用 AngularJS 验证机制
  • Java 领域对象，使用 Bean 验证

然后，Bean 验证将用于在域对象使用时自动验证：

  • Spring MVC REST controllers (使用 @Valid 注解)
  • Hibernate/JPA（实体在保存之前自动验证）

验证信息还将用于生成更精确的数据库列元数据：

  • 必填字段将被标记为不可为空（non-nullable）
  • 设置最大长度的字段将具有相同的列长度

验证有一些限制：

  • 我们不支持 AngularJS 和 Bean 验证的所有验证选项，因为我们只支持这两种 API 通用的验证选项
  • 正则表达式模式不能同时在 JavaScript 和 Java 中起作用，因此，如果您配置一个正则表达式模式，则可能需要调整生成的正则表达式模式
  • JHipster 生成可用于通用实体的单元测试，而不知道您的验证规则：生成的测试可能未通过验证规则。 在这种情况下，您将需要修改单元测试中使用的示例值，以便它们通过验证规则。

更新已存在的 Entity

实体配置保存在.jhipster目录中的特定.json文件中。 因此，如果再次运行子生成器，则使用现有实体名称，可以更新或重新生成实体。

当您为现有实体运行 entity sub-generator 时，将会询问您是否要更新实体？ 这将替换此实体的现有文件，所有您的自定义代码将被覆盖具有以下选项：

  • Yes, re generate the entity - 这只会重新生成您的实体。 提示：运行子生成器时可以通过传递一个 --regenerate 标志来强制执行此操作
  • Yes, add more fields and relationships - 这将给您提供更多的字段和关系的提问
  • Yes, remove fields and relationships - 这将为您提供问题以从实体中删除现有的字段和关系
  • No, exit - 这将退出子生成器而不改变任何东西

您可能需要更新您的实体，原因如下：

  • 您想要添加/删除现有实体的字段和关系
  • 您要将实体代码重置为原始状态
  • 您已更新 JHipster，并希望通过新模板生成实体
  • 您已经修改了.json配置文件（格式与生成器提出的问题非常接近，所以不是很复杂），因此您可以拥有一个新版本的实体
  • 你已经拷贝/粘贴了.json文件，想要一个非常接近复制实体的新实体

提示：要一次重新生成所有实体，您可以使用以下命令（删除--force在文件更改时询问问题）。

  • Linux & Mac: for f in `ls .jhipster`; do jhipster entity ${f%.*} --force ; done
  • Windows: for %f in (.jhipster/*) do jhipster entity %~nf --force
